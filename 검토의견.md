# 🔍 프로젝트 전체 검토 의견

> **검토일**: 2026-07-03 · **대상**: 몬스터 헌터 러너 전체 (src 전 파일 정독 + CDP 런타임 검증)
> **검증 방법**: 소스 코드 전량 리뷰 + 실행 중인 게임에 CDP(127.0.0.1:9222) 연결하여 시나리오 재현·스크린샷·FPS/드로우콜 계측. **본 문서는 검토 의견만 담으며 코드는 일절 수정하지 않았다.**

## 총평

게임플레이 루프·상태머신·데이터 구동 설계(월드/보스/몬스터 전부 `worlds.ts`)·view-logic 분리·접근성(입력 관용, 자동 스킬)은 **탄탄하고 설계서와 높은 정합성**을 보인다. 57fps, 드로우콜 89, 삼각형 1.7K로 성능 여유도 충분. 다만 **게임 디자인을 실질적으로 깨는 버그 1건(P0)**과 모바일 배포 전 반드시 손봐야 할 리소스/수명주기 이슈가 있다.

## 1. 버그 (심각도순)

### 🔴 P0 — 보스 `walls` 패턴이 데미지를 절대 못 준다 (런타임 확정)

- **위치**: `entities/Boss.ts` — `beginActive()`의 walls 생성부와 `updateHazards()`의 판정부.
- **원인**: 벽은 `player.z + 1.5`에 생성되는데, 접촉 판정은 `Math.abs(player.z - wall.z) < 1.0`. 보스전에서는 전진 정지(`runSpeed=0`)라 플레이어 z가 불변 → **dz는 영원히 1.5, 판정 임계 1.0을 절대 못 넘음**.
- **재현 증거**: 괴담 소녀 `hair` 패턴 중 플레이어를 벽 레인(L1)에 강제 고정하고 6틱 관찰 → `wall(z1.5,L1) p(z0,L1)` 상태에서 HP 85 유지(무피해). 같은 세션에서 scream은 정상적으로 15 데미지.
- **영향**: walls를 쓰는 보스 6종(괴담 소녀·좀비 킹·크라켄·드라큘라·해골 기사 + 박쥐 떼 벽)의 핵심 위협이 **장식**이 됨. "안전 레인 이동" 회피 학습 루프 전체가 무의미해지고, 설계서 §9의 예상 전투 시간·데미지 테이블(머리카락 20 등)과 실제 난이도가 괴리.
- **제안**: 판정 임계를 생성 오프셋과 일치시키거나(예: `< 1.8`), 벽 z를 `player.z + 0.5`로 당기거나, 레인 점유 기반 판정(벽 지속 중 같은 레인이면 주기 데미지)으로 교체. 수정 후 §9 예상 전투 시간 재검산 필요.

### 🟠 P1 — 백그라운드 탭에서 게임이 '슬로모션'으로 계속 진행됨 (런타임 확정)

- **위치**: `core/Game.ts` 게임루프(rAF + frameDt 0.1s 클램프), visibilitychange 미처리.
- **증상**: 탭이 백그라운드면 rAF가 ~1fps로 스로틀 → 프레임당 게임시간 0.1s만 진행되어 **약 10% 속도의 슬로모션으로 계속 플레이됨**. 관찰: 백그라운드에서 보스가 0.4s짜리 `reappear` 상태에 8초 이상 정체, 그 사이 피격·자동사격도 왜곡.
- **영향**: 모바일(홈 이동/알림)에서 복귀 시 이미 죽어있거나 상태가 뒤틀릴 수 있음. 어린이 타깃에서 특히 불리.
- **제안**: `document.visibilitychange` → hidden 시 자동 일시정지(`togglePause` 재사용). 3~5줄 수정.

### 🟠 P1 — REWARD 화면 뒤로 HUD가 그대로 노출 + 일시정지 버튼 무반응

- **증거**: 보상 화면 스크린샷에서 좌상단 체력바·레벨, 우하단 스킬 버튼(AUTO 배지 포함)이 보상 연출과 겹쳐 표시됨.
- **원인**: `setState('REWARD')`에서 `hud.hide()` 미호출(RESULT/GAMEOVER만 숨김). 또한 REWARD는 `isPlayState`가 아니라 HUD의 ⏸ 버튼이 눌러도 무반응.
- **제안**: REWARD 진입 시 HUD 숨김(또는 스킬/일시정지 버튼만 숨김).

### 🟡 P2 — 사운드 배선 5종 누락: 문서 주장과 불일치

- `SoundId` 25종 중 **`laneMove` / `jump` / `slide` / `bgmRun` / `bgmBoss` 5종은 어떤 코드에서도 `play()` 되지 않음** (전수 검색으로 확인). 특히 **BGM은 재생 경로 자체가 없음**.
- `에셋소싱.md` §0.4·§9는 "호출 지점은 전 코드에 배선 완료 — 파일만 채우면 됨"이라고 명시 → **거짓 문장**. M4(사운드)에서 파일만 넣으면 끝나는 게 아니라 배선 작업이 추가로 필요.
- **제안**: 지금 단계에서는 문서를 사실대로 고치거나(5종 미배선 명기), M4 전에 배선 추가. Player의 점프/슬라이드/레인이동 성공 지점과 상태 전환부(BGM)가 삽입 위치.

### 🟡 P2 — favicon 404

- `index.html`에 파비콘 미지정 → 최초 로드 시 404 1건(콘솔 오염). 한 줄로 해결 가능.

### ⚪ P3 — 잠재 결함 (실해는 미미, 인지만)

- **Boss barrage 종료 이중 경로**: active에서 `state='recovery'` 설정 후에도 같은 프레임에 `timer<=0`이면 `afterPattern()`이 또 탈 수 있는 구조. 현재 수치(탄막 ≤1.5s, 안전상한 6s)에서는 발생 불가하지만, 데이터로 `shots×interval > 6s` 보스를 추가하면 터짐.
- **coyote 타임이 `performance.now()` 기준**: 일시정지 중에도 실시간이 흘러 공중에서 일시정지→재개 시 코요테 판정 소실. 체감 거의 없음.
- **동일 오리진 잔재 키**: localhost:5173 localStorage에 타 프로젝트 `mmr_save` 키 존재. `mhr_` 프리픽스 덕에 충돌은 없음(정보성).

## 2. 성능 / 모바일 배포 준비

### 🟠 이미지 총 ~8.7MB를 부팅 시 전부 프리로드

- `public/assets/images` 13장(타이틀 배경 920KB, 월드 6장 각 540~774KB, 보상 6장 각 560~740KB)을 `Screens` 생성자에서 **일괄 `new Image()` 프리로드**.
- 중급폰·모바일 데이터에서 초기 로딩과 트래픽 부담이 큼. 설계서 성능 예산(§에셋소싱 0.1)에 견줘 과함.
- **제안**: ① WebP/AVIF 변환 + 실표시 크기(월드 썸네일 ~256px, 인트로 ~1024px)로 리사이즈 → 10~20배 절감 가능. ② 프리로드는 타이틀 배경+현재 월드만, 나머지는 지연 로드.

### 🟡 three.js 단일 청크 583KB(gzip 151KB)

- 빌드 경고 발생 중. 게임 특성상 어차피 전부 필요해 우선순위는 낮지만, 로딩 스피너 없이 흰 화면이 뜨는 시간이 생김. **제안**: 초기 로딩 인디케이터(HTML 단독) 추가가 코드 스플리팅보다 실익 큼.

### 🟡 GPU 리소스 dispose 누락 (장시간 세션 누적)

- 보스 `buildBody()`가 보스전마다 지오메트리·머티리얼을 새로 생성하고 `dispose()`에서 scene 제거만 함(`geometry.dispose()`/`material.dispose()` 미호출). wave 머티리얼, blast 링 clone도 동일.
- 한 판 규모에선 무해하나 연속 플레이(6월드 연속×반복)에서 GPU 메모리 누적. **제안**: `Boss.dispose()`에서 traverse하며 dispose, 이펙트류는 재사용 풀.

### ✅ 양호한 항목 (계측치)

- 포그라운드 57~61fps, 드로우콜 89, 삼각형 1,706, pixelRatio 클램프(≤2) — 예산 대비 여유 큼.
- 지오메트리/머티리얼 모듈 레벨 공유·캐시(Monster/Obstacle/Pickup/Projectile) — 잘 되어 있음.
- 고정 타임스텝 + 프레임 독립 lerp 카메라 — 안정적.

## 4. 게임 디자인 관찰 (밸런스·UX)

1. **방치(자동 스킬만) 시 생존력**: 자동 대시는 장애물·투사체만 감지하고 레인 회피는 안 하므로 방치 시 HP가 빠르게 소모(관찰: 러닝 ~30초 방치에 100→35). PIT 데미지 30이 특히 아픔. 6~12세 타깃 기준 "자동만 믿는 아이"의 좌절 가능성 — PIT만 자동 대시 트리거에 점프 대응을 추가하거나 PIT 데미지 하향 검토 여지.
2. **scream 회피 판정 매우 관대**: active 중 아무 때나 1프레임이라도 슬라이드하면 회피 성사(래치). 어린이용으로는 적절해 보임 — 의도 명시만 문서에 남기면 좋음.
3. **walls 수정 시 난이도 급상승 주의**: P0 수정 순간 보스 6종의 실효 난이도가 설계값으로 '복원'됨. 어린이 플레이테스트 없이 바로 올리면 급격히 어려워질 수 있어, 수정과 함께 벽 데미지/지속시간 하향 여지를 둘 것.
4. **튜토리얼 스킬 단계가 해금 스킬을 다루지 않음**: R/F 스킬은 해금 후 별도 안내가 REWARD 문구 한 줄뿐. 3월드 클리어 직후 다음 판 시작 시 짧은 배너("R키/🔥 버튼 사용 가능!") 정도가 있으면 발견성이 좋아짐.
5. **P4(2레인 차단) 안전 레인이 플레이어 위치와 무관하게 랜덤**: 항상 통과는 가능하지만 최악의 경우 2레인 연속 이동이 필요. `buildPattern`이 `playerLane`을 받고도 미사용(`void playerLane`) — 후반 간격 1.2s에서는 문제없으나 의도적 미사용이면 주석으로 남길 것.
6. **터치 UX**: 중앙 탭=점프 규칙이 스킬 버튼·일시정지와 겹칠 수 있으나 버튼이 `preventDefault`로 선점 — 동작 확인됨. 다만 화면 하단 1/3 스와이프 유도(설계서 §5)와 달리 전체 화면이 입력 영역인 점은 문서와 미세 불일치(실사용엔 오히려 유리).

## 5. 코드 품질 / 유지보수

1. **`core/Game.ts` 비대(약 880줄)**: 상태머신 + 저장/이관 + 스폰 헬퍼 + 체크포인트 + 환경까지 한 파일. `Persistence`(storage 계층)와 `Environment`(이미 클래스 분리돼 있음 → 별도 파일) 분리만 해도 가독성이 크게 개선.
2. **테스트 부재**: 순수 로직(스킬 해금 판정, 레거시 이관, 패턴 안전경로, 페이즈 전환 인덱스, 점수 공식)이 전부 테스트 가능한 형태인데 러너가 없음. vitest 도입 + 위 5개 유닛만 깔아도 밸런스 리팩터링 안전망이 생김. 특히 `pickLanes()`(항상 안전 레인 ≥1 보장)와 `buildPattern()`은 속성 테스트 대상으로 최적.
3. **린트/포맷 설정 부재**: eslint/prettier 없음. 1인 개발이라도 CI 없이 tsc만으로는 `void playerLane` 같은 흔적이 남음.
4. **타입 캐스트 정리 여지**: `CONFIG.skills.unlocks as Partial<Record<SkillId, number>>`, `btn.dataset.skill as SkillId` 등 — `unlocks`에 `satisfies Partial<Record<SkillId, number>>`를 붙이면 소비측 캐스트 제거 가능.
5. **매직 넘버 일부 산재**: 히트 판정 반경(0.75/1.4/0.9/0.95), 보스 아레나 y, 스테이지 인트로 3.0s 등이 코드에 직접 존재. config 단일 소스 원칙(설계서 §20)을 따르면 `CONFIG.combat.hitRadius` 류로 승격 대상.
6. **`window.game` 전역 노출**: E2E용으로 유용(이번 검토에서도 활용). 프로덕션 빌드에서 치트 경로가 되지만 로컬 게임이라 무해 — `import.meta.env.DEV` 가드 정도만 고려.
